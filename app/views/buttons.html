Buttons

Buttons allow users to take actions which affect their current task.

Usage Guidelines

When to use

  • Buttons can be used to start a new task, such as signing up for a service or registering a new account.
  • Buttons can be used to impact a task the user is currently working on, such as clearing a form or saving a draft.
  • Buttons can be used to finish a task, such as submitting a form.

When not to use

  • Does the user need to navigate to another page? Consider using a navigation component, like a link or Back/Next icons instead (e.g. to save space).
  • Does an icon exist which represents the action? If so, you may wish to use the appropriate icon instead (e.g. to save space).

Interaction

Buttons generally fall under the categories of “primary” and “secondary”, each of which have different designs/visual weight. Which button you use depends on whether the associated action is primary or secondary:

Primary Actions

Active

Primary actions are those in which a user is starting a new process (e.g. “Register”), progressing/adding to their current one (e.g. “Add Image”), or completing a process (e.g. “Submit”).

Primary buttons have static, hover, and active states, the latter two of which are darker in color.

Disabled

For cases in which a primary action is disabled (e.g. they must complete part of a form before submitting), a disabled option should be used. Disabled buttons are also paired with a disabled mouse icon.

Secondary Actions

Guidelines

Secondary actions are those within an existing process which are less frequently used. They often take the form of removing data (e.g. “Delete”) or stopping/resetting a process (e.g. “Cancel”).

Like primary buttons, secondary actions also have static, hover, and active states.

Presentation

Make sure there is enough visual contrast for the button to stand out from other elements on the page and to represent the importance of the action. This is particularly true when a button causes an irreversible action—in which case, you may wish to also include a confirmation modal.

Labels

Guidelines

Button labels should be short and actionable, and clearly explain the action that will take place when it is selected. The text on a button should be two words or less whenever possible. The button’s physical placement (and its placement within a workflow) should provide enough context for a user that lengthy descriptive text isn’t necessary.

Placement

Button placement can vary quite a bit depending on the situation, but there are some thing to consider:

  • What is the user’s workflow across a page? Primary buttons in particular should be logically placed according to where the user is going next. For example, if a user is filling out a form, placing the submit button near the last item for them to complete is sensible.
  • Is a button going to be used frequently during a workflow? If so, it should be persistent in the user’s field of vision (e.g. pinning it in the navigation menu).
  • Do you need to prevent accidental clicks? In the case of irreversible actions in particular, you may wish to space it away from other interactive elements (e.g. separating out a “Delete” and “Save” button).

Hierarchy of Actions

It is possible that several primary and secondary actions will be available to a user within a given page. By considering several factors, you can establish a hierarchy between actions. This will help you determine which type of button you use and/or their placement relative to each other.

  • Importance of action: Is an action very important in a workflow (e.g. complete a process, submit an important form, etc.)?
  • Frequency of use: Is the action used very frequently?
  • Potential harm: Could the action be very harmful to your user’s workflow? In this case, you may wish it to stand out less than the safe action.
  • Opposing functions: Are any of the buttons opposites (e.g. “Delete” and “Save”)?
